# 运行模式与斟酌项

这份文档聚焦“有哪些运行路径、每条路径适合什么、权衡在哪里”。

## 1. 运行模式不是只有一种

这个仓库目前支持四类运行模式。

| 模式 | 目标 | 是否依赖教学 VM | 是否依赖 archived runs | 适合谁 |
| --- | --- | --- | --- | --- |
| Mode A: Reverse-SSH Direct Runner | 直接执行实验 | 是 | 否 | 当前课程环境的主线使用者 |
| Mode B: seed-runner Session Runner | 通用化远端执行 | 是 | 否 | 想扩展到多机/持久 session 的使用者 |
| Mode C: Offline Report Packaging | 只做报告与证据整理 | 否 | 是 | 想离线验证报告链或只处理已有结果的人 |
| Mode D: MASFactory Studio | 展示与统一入口 | 视实际触发的主线能力而定 | 可选 | 想做 demo / 可视化工作台的人 |

## 2. Mode A: Reverse-SSH Direct Mode

这是当前主线和默认推荐。

### 特点

- 假设 reverse SSH 已经打通
- 直接面向当前 Proxmox 教学 VM
- 路径最短
- 最贴近当前课程真实环境

### 优点

- 快
- 简洁
- 已在当前课程环境中验证
- 最少引入额外工具

### 风险

- 依赖反向隧道是活的
- 更偏单机，不够通用
- 会把课程环境假设带得比较深

### 什么时候用

- 当前课程作业主线
- 已确认 `ssh -i ~/.ssh/seed-way -p 2345 seed@localhost` 可用
- 想尽快把实验跑完

## 3. Mode B: seed-runner Session Mode

这是备选的更通用远端执行面。

### 特点

- 基于 `sshfs` + `tmux` + session exec
- 适合多机器、共享挂载、会话持久化

### 优点

- 更一般化
- 更适合多机
- 更适合远端 session 级记录和维护
- 更容易演化成平台型远端执行工具链

### 风险

- 接入成本更高
- 需要额外配置 `.env.machines`
- 需要更多前置环境准备

### 什么时候用

- 不想把系统绑定在当前 reverse-SSH 上
- 有多台实验机
- 想做更泛化的远端执行架构

## 4. Mode C: Offline Report Packaging

这是最容易先跑通的一条链。

### 特点

- 不重新做实验
- 只消费 `reports/`
- 可以离线验证 skill 和报告效果

### 优点

- 没有远端依赖
- 适合让另一个智能体先安装并验证仓库
- 适合作为 smoke test

### 风险

- 它只能证明“报告链是通的”
- 不能证明“实验执行链一定通”

### 什么时候用

- 刚拿到仓库
- 教学 VM 暂时不可用
- 想先确认打包和证据层稳定

## 5. Mode D: MASFactory Studio

这是展示层，不是主线执行引擎。

### 特点

- 统一入口
- workflow 可视化
- dashboard / PDF / Manual UI 聚合

### 优点

- 更适合展示
- 更适合作为“对外演示的工作台”
- 让整个系统更像一个产品，而不是脚本集合

### 风险

- 比主线多一层包装
- 如果过早依赖它，容易模糊主线边界
- 不适合作为新增实验的第一落点

### 什么时候用

- 主线已经稳定
- 想做 demo
- 想让另一个智能体从更清晰的统一入口开始理解系统

## 6. 实际推荐顺序

如果是第一次接触这套仓库，推荐路线不是“从最炫的开始”，而是：

1. 先做 `Mode C`
2. 再做 `Mode A`
3. 需要更泛化时考虑 `Mode B`
4. 需要展示时再接 `Mode D`

这样做的原因是：

- 先验证报告链最稳
- 再验证真实实验链
- 最后再接泛化层和展示层

## 7. 如何选择

可以用这张判断表：

| 你的目标 | 推荐模式 |
| --- | --- |
| 我只想确认仓库和 skill 能不能先跑起来 | Mode C |
| 我想在当前课程环境里真正做实验 | Mode A |
| 我想把远端执行做得更通用、更适合多机 | Mode B |
| 我想做统一交互入口和 demo | Mode D |
